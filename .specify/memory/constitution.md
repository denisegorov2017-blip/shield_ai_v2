# Конституция проекта shield_ai_v2

## Основные принципы

### Библиотечный подход

Каждая функция начинается как автономная библиотека; Библиотеки должны быть автономными, независимо тестируемыми, документированными; Требуется четкая цель - без библиотек только для организации.

### Интерфейс командной строки

Каждая библиотека предоставляет функциональность через CLI; Протокол текстового ввода/вывода: stdin/args → stdout, ошибки → stderr; Поддержка JSON и человекочитаемых форматов.

### Тестирование первым (НЕОБСУЖДАЕМОЕ)

TDD обязательно: Сначала тесты → одобрение пользователем → тесты не проходят → затем реализация; Строгое соблюдение цикла Red-Green-Refactor.

### Интеграционное тестирование

Области, требующие интеграционных тестов: Тестирование контрактов новых библиотек, изменения контрактов, межсервисное взаимодействие, общие схемы.

### Наблюдаемость, Версионирование и критические изменения, Простота

Текстовый ввод/вывод обеспечивает отладку; Требуется структурированное ведение логов; Или: формат MAJOR.MINOR.BUILD; Или: начинать с простого, принципы YAGNI.

## Дополнительные ограничения

Требования к технологическому стеку, стандарты соответствия, политики развертывания и т.д.

## Рабочий процесс разработки

Требования к код-ревью, тестовые ворота, процесс утверждения развертывания и т.д.

## Управление

Конституция превалирует над всеми другими практиками; Изменения требуют документирования, утверждения, плана миграции. Все PR/ревью должны проверять соответствие; Сложность должна быть обоснована; Используйте .kilocode/rules/Ruls_prodject.md для руководства при разработке во время выполнения.
